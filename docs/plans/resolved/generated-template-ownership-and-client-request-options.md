# Generated Template Ownership And Client Request Options

This plan records two follow-up design decisions before implementation:

- Template source files should follow the same ownership convention as generated files.
- Generated clients should expose per-call request options in a language-native way.

This file started as a design record. The plan was approved and implemented on
2026-05-21; it now records the accepted contract and the validation scope that
closed the implementation.

## Implementation Status

- Status: implemented.
- Date: 2026-05-21.
- Scope: generated template ownership naming, client per-call request options,
  docs, tests, and refreshed blueprint examples.
- Validation: see the Test Plan commands at the end of this document.

## Goals

- Make generator-owned templates visibly generator-owned before reading writer code.
- Avoid hidden ownership contracts where a template has a preserved-looking name but renders an overwritten `Gen*` / `gen_*` artifact.
- Provide per-call headers, timeout, and request customization for client RPC calls across supported HTTP/Wails clients.
- Keep long-lived stream/channel lifecycle separate from unary RPC timeout semantics.
- Document gRPC as a protocol-native exception: deadlines and call options belong to generated gRPC clients, not to api-blueprint proto output.

## Template Ownership Invariant

All templates used to render generator-owned, overwritten, or generated-header artifacts must satisfy at least one ownership marker:

- File name starts with `gen_`.
- File name starts with `Gen`.
- The template is under a `_gen_*` directory.

Preserved scaffold, facade, implementation, service, and user-entry templates may keep non-Gen names, but they must not contain a `Code generated by api-blueprint ... DO NOT EDIT` header.

Generated output remains the source of user-facing ownership, but template names must now match the same mental model. A reader should not need to inspect writer `overwrite=True` calls to know whether a template is generator-owned.

## Client Request Options Model

Per-call request options are a client SDK concern, not a DSL route property. They should not alter route contracts, server behavior, or generated schema shape.

The minimum cross-target capability is:

- Per-call headers.
- Per-call timeout for unary RPC / invoke / HTTP request.
- Global default headers and global default timeout where the transport has a natural config object.
- Native request hook only where idiomatic and already supported by the target.

Header precedence is:

1. Transport/global default headers.
2. Per-call request options.
3. Body-driven protocol headers such as `Content-Type` when the transport must set them for JSON, form, multipart, or binary bodies.

Per-call timeout precedence is:

1. Per-call request option.
2. Transport/global default timeout.
3. Native client default.

Stream and channel timeout remains out of scope for this request. Their lifecycle is controlled by context cancellation, close APIs, socket/event runtime behavior, or app code.

## Client Options Matrix

| Target | Current state | Planned API | Transport behavior |
| --- | --- | --- | --- |
| TypeScript HTTP | Already supports headers, `RequestInit`, and `timeoutMs`, but route signatures expose `init` and `timeoutMs` as loose trailing parameters. | Introduce `ApiRequestOptions` and change route methods to `method(request, options?: ApiRequestOptions)`. | Merge `options.headers`; pass `options.init`; use `options.timeoutMs ?? config.timeoutMs`. |
| TypeScript Wails | Shared request options exist; headers enter Wails envelopes. | Reuse `ApiRequestOptions` shape for Wails route methods. | Headers remain envelope metadata; `timeoutMs` wraps the frontend promise and does not cancel the native Wails call. |
| Java HTTP | `GenApiRequest` has headers internally, but route methods pass `Map.of()` and config has only `baseUrl`. | Add `GenApiRequestOptions` with builder helpers; route methods get overloads with options. | Merge default headers and options headers; apply `HttpRequest.Builder.timeout(options.timeout())` when set. |
| Kotlin HTTP | Route methods accept headers; config has default headers; no per-call timeout. | Add `ApiRequestOptions(headers, timeout)` and use `options: ApiRequestOptions = ApiRequestOptions()` in route methods. | Merge default headers and options headers; use per-call OkHttp `callTimeout` when set. |
| Flutter HTTP | Route methods accept headers; config has global timeout; no per-call timeout/options object. | Add `ApiRequestOptions(headers, timeout)` and use named `options` in route methods. | Merge default headers and options headers; use `options.timeout ?? config.timeout`. |
| Python HTTP | Transport protocol can receive route fields, but route RPC methods do not expose headers/timeout. | Add keyword-only `headers: Mapping[str, str] | None = None` and `timeout: float | None = None` to RPC methods. | Pass headers and timeout to `httpx.AsyncClient.request`; advanced users keep injecting a configured `httpx.AsyncClient`. |
| Go HTTP | Route methods take `context.Context`, which already supports deadline/cancel; request headers are not exposed. | Add variadic `runtime.RequestOption` to route methods, with `WithHeader` and `WithHeaders`. | Apply options to runtime request headers; merge config default headers before per-call headers; timeout remains `context.Context`. |
| gRPC proto | api-blueprint only writes proto files and optional protoc outputs. | No api-blueprint client options wrapper. | Document native gRPC deadlines, metadata, interceptors, and call options in generated-client docs. |

## Target API Examples

### TypeScript HTTP

```ts
await client.demo.preview(
  { multipart },
  {
    headers: { "X-Trace-Id": traceId },
    timeoutMs: 3000,
    init: { credentials: "include" },
  },
);
```

### TypeScript Wails

```ts
await client.demo.preview(
  { json: payload },
  {
    headers: { "X-Trace-Id": traceId },
    timeoutMs: 3000,
  },
);
```

`timeoutMs` rejects the frontend promise if the native call does not finish in time. It does not cancel the underlying Wails invoke.

### Java HTTP

```java
var result = api.preview(
    form,
    GenApiRequestOptions.builder()
        .header("X-Trace-Id", traceId)
        .timeout(Duration.ofSeconds(3))
        .build()
);
```

Existing route methods without options remain as convenience overloads.

### Kotlin HTTP

```kotlin
val result = api.preview(
    form,
    options = ApiRequestOptions(
        headers = mapOf("X-Trace-Id" to traceId),
        timeout = 3.seconds,
    ),
)
```

`ApiRequestOptions` should use Kotlin idioms and avoid Java builder style.

### Flutter HTTP

```dart
final result = await api.preview(
  multipart: form,
  options: ApiRequestOptions(
    headers: {'X-Trace-Id': traceId},
    timeout: const Duration(seconds: 3),
  ),
);
```

The old loose `headers:` route parameter should be replaced by `options.headers`.

### Python HTTP

```python
result = await api.preview(
    form=form,
    headers={"X-Trace-Id": trace_id},
    timeout=3.0,
)
```

`timeout` is passed to httpx per request. Users that need a richer `httpx.Timeout` policy can inject a configured `httpx.AsyncClient`.

### Go HTTP

```go
result, err := api.Preview(
	ctx,
	form,
	runtime.WithHeader("X-Trace-Id", traceID),
)
```

Timeouts continue to use context:

```go
ctx, cancel := context.WithTimeout(ctx, 3*time.Second)
defer cancel()

result, err := api.Preview(ctx, form, runtime.WithHeader("X-Trace-Id", traceID))
```

### gRPC

For generated gRPC clients, use native metadata and deadlines:

```go
ctx, cancel := context.WithTimeout(ctx, 3*time.Second)
defer cancel()

ctx = metadata.AppendToOutgoingContext(ctx, "x-trace-id", traceID)
result, err := client.Preview(ctx, request)
```

api-blueprint proto generation should not invent an additional cross-language client options layer.

## Compatibility Notes

- TypeScript, Kotlin, and Flutter route APIs intentionally move from loose headers/timeout parameters toward a single options object. This is a breaking route-call-shape cleanup.
- Java should minimize breakage by adding overloads instead of removing existing methods immediately.
- Python keeps idiomatic keyword-only parameters rather than introducing a heavy options dataclass.
- Go keeps timeout on `context.Context` and adds variadic options for headers and future request-level flags.
- gRPC output remains proto-first; timeout/header behavior is documented as native gRPC client usage.

## Acceptance Criteria

- The plan was reviewed and approved before implementation started.
- Template ownership and generated artifact ownership have matching tests.
- Every HTTP client target has an explicit per-call way to set request headers and timeout.
- Go documents timeout through `context.Context` and adds request options for headers.
- Wails documents timeout as frontend waiting behavior, not native cancellation.
- gRPC documents native metadata/deadline paths instead of implying a missing api-blueprint capability.
