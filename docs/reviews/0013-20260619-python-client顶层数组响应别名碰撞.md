# python client 顶层数组响应别名碰撞

## 状态

待修复，继续跟踪。

## 发现日期

2026-06-19

## 背景

真实接口 `/base/perm/list` 返回标准 envelope，`data` 是顶层数组：

```json
{
  "code": 200,
  "msg": "",
  "data": [
    { "id": 1, "name": "...", "code": "..." }
  ]
}
```

DSL 使用 `RSP(Array[AdminMBasePermission]())` 表达顶层数组响应。预期 Python client route 返回 `list[AdminMBasePermission]` 并使用 `_decode_list(..., AdminMBasePermission.from_value)` 解码。

## 风险分级

高。问题会导致 Python generated client 对部分真实接口生成错误返回类型和错误解码逻辑，运行时无法正确消费服务端返回的数组数据。

## 问题性质

ContractGraph schema identity / alias collision 缺口，并在 Python client 生成层表现为错误 response annotation 和错误 decoder。

## 存在性判断

已确认。

`Router._RSP_AS_MODEL` 会把 `RSP(Array[Model]())` 这类 Field response 包成 `FieldWrappedModel`，名字来自 `RSP_{route.name}`。多个 route leaf 都是 `list` 时会产生相同的 `RSP_List`。

普通 auto response model 会走 `ContractGraphBuilder._schema_id` 的同名冲突处理；但 `FieldWrappedModel` / `Field` alias 分支只用短名 `RSP_List`，没有处理已有 schema 同名但语义不同的情况。如果某个普通 `RSP_List` object schema 先注册，后续 `RSP(Array[...])` alias 会直接复用已有 `RSP_List`，导致数组 route 的 response manifest 指向对象 schema。

Python client 随后从 manifest 读取 response schema。如果 `RSP_List` 已是 object schema，`PythonSchemaRegistry.resolve_schema(..., class_name="ListResponse")` 会生成 `ListResponse.from_value(...)`，而不是 `list[AdminMBasePermission]` 的 `_decode_list(...)`。

## 复现场景

最小复现：

```python
from api_blueprint.contract import build_contract_graph
from api_blueprint.engine import Blueprint, Model
from api_blueprint.engine.model import Array, Int64, String

class Permission(Model):
    id = Int64(description="id")
    name = String(description="name")
    code = String(description="code")

bp = Blueprint(root="/api")

with bp.group("/base/role") as role:
    role.GET("/list").RSP(total=Int64(description="total"))

with bp.group("/base/perm") as perm:
    perm.GET("/list").RSP(Array[Permission](description="permissions"))

manifest = build_contract_graph([bp]).to_manifest()

assert manifest["routes"][0]["response"]["model"] == "RSP_List"
assert manifest["routes"][1]["response"]["model"] == "RSP_List"
assert manifest["schemas"]["RSP_List"]["type"] == "object"
```

生成 Python client 后，两个 route 都会出现类似输出：

```python
async def list(...) -> ListResponse:
    response_type: str | None = "ListResponse"
    ...
    return ListResponse.from_value(payload, "list.response")
```

其中 `/base/perm/list` 本应生成：

```python
async def list(...) -> list[Permission]:
    response_type: str | None = "list[Permission]"
    ...
    return _decode_list(payload, "list.response", Permission.from_value)
```

## 影响范围

- Python client 受影响最明显，会生成错误类型和错误 decoder。
- ContractGraph manifest 本身已经错误，因此其他依赖 full manifest 或 shards 的插件/target 也可能受到影响。
- 高风险场景是多个 group 下存在相同 leaf / operation，例如 `/base/role/list`、`/base/perm/list`、`/xxx/list`，其中至少一个 route 使用 `RSP(Array[...])` 或其他 FieldWrapped response。
- 如果数组 alias 先注册、后续普通 object model 再注册，普通 model 的 `_schema_id` 可能会把已有 alias 改写到 qualified identity；这个路径需要一并测试，避免只修单向顺序。

## 兼容性 / 修复风险

修复应优先放在 ContractGraph 层，而不是 Python client 模板层兜底。生成器应保证 manifest 中每个 schema ref 指向语义正确的 schema。

可行方向：

- 让 `FieldWrappedModel` / `Field` alias schema 使用与普通 model 一致的身份和冲突处理策略。
- 或为 auto alias 生成 route-scoped / identity-qualified schema id，确保 `RSP_List` 短名碰撞时能改写到稳定唯一 key。
- 修复时必须同步更新已有 route/schema ref rewrite，覆盖 routes、schemas、exported_models 等 manifest 区域。

兼容性风险是 manifest schema key 可能从短名变为 qualified identity；但这与普通 model 同名冲突策略一致，属于修复错误引用而不是改变正常无冲突输出。

## 是否建议修复

建议修复，并在 `v2.0.6` 发版前处理。该问题已对应真实接口生成不可用 client，属于协议契约和生成物可用性问题。

## 后续处置建议

- 增加 ContractGraph 测试：普通 `RSP_List` object 与 `RSP(Array[Permission]())` 同名时，两个 route response schema 必须分离且数组 route schema target 为 array。
- 增加 Python client codegen 测试：`/base/perm/list` 生成 `list[Permission]` annotation 和 `_decode_list(..., Permission.from_value)`，不生成 `ListResponse.from_value`。
- 覆盖注册顺序的两个方向：object 先、array 后；array 先、object 后。
- 修复完成后移动到 `docs/reviews/resolved/`，追加 Resolution、验证命令以及相关 commit/PR。
