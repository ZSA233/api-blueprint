# OpenAPI 不承载消息协议设计边界

- 状态：基础能力已实施；interaction-first UI 与 metadata/plugin projection 已实施；try-out console / upstream codec 仍需后续设计
- 发现日期：2026-06-22
- 背景：用户反馈 `app_socket`、op/message/channel/custom transport 不应伪装成 OpenAPI HTTP endpoint。OpenAPI 的核心模型是 HTTP path/method/request/response，强行映射消息协议会误导文档读者、H5 调试和客户端接入。
- 风险分级：中
- 问题性质：协议文档边界与用户体验风险
- 存在性判断：真实存在。docs center 需要把 `STREAM` / `CHANNEL` 和 op/message catalog 作为原生消息协议展示，而不是依赖 Swagger/OpenAPI。

## 风险说明

- 把 WebSocket / app socket op 伪装成 HTTP path 会造成错误的调用模型。
- 纯消息协议服务如果只依赖 Swagger/OpenAPI，会缺少 op 编号、消息方向、push model、close model、frame metadata 和示例 payload。
- try-out console 若默认直连生产 upstream，会引入 token 暴露、frame codec 猜测、权限绕过和误操作风险。

## 复现场景

1. 定义只包含 `CHANNEL` 的 Blueprint。
2. 用户打开 Swagger 或消费 `/openapi.json`。
3. 若将 channel 映射成假 HTTP endpoint，用户会误以为存在普通 HTTP request/response。
4. 若完全不展示 channel 详情，用户又无法从 docs 中理解 op/message 协议。

## 影响范围

- docs center / docs server。
- `STREAM` / `CHANNEL` / custom message protocol 的文档体验。
- 外部工具消费 `/openapi.json`、`/docs/protocol.json`、`/asyncapi.json` 的语义边界。

## 兼容性与修复风险

- 保持 `/openapi.json` 只表达 OpenAPI 能表达的 HTTP surface，兼容现有 OpenAPI 工具。
- 新增 protocol catalog 和 AsyncAPI 是增量入口，不破坏现有 HTTP docs。
- AsyncAPI 是导出视图，不作为 api-blueprint 内部真源，避免受标准模型限制反向污染 DSL。
- 不默认实现可连接 console，避免安全边界扩大。

## 是否建议修复

建议修复。优先新增原生 protocol catalog 和 AsyncAPI 导出；try-out console 只保留占位，后续另行设计项目自有 upstream/codec 扩展。

## 后续处置建议

- 已实施基础方向：`/docs/protocol`、`/docs/asyncapi`、`/docs/protocol.json`、`/asyncapi.json`，并在 docs center 为 `STREAM` / `CHANNEL` 提供专用 Protocol UI / AsyncAPI UI 入口。
- 已补 interaction projection 边界：默认 `CHANNEL` 不猜测请求/响应配对；内置 metadata interaction 插件读取 `message_variant(..., interaction=..., role=...)` 生成请求/响应关联；项目可通过 `[blueprint].protocol_docs_plugins` 加载自定义 projection 插件，把 app socket、op 表或业务 metadata 投影到同一套 interaction catalog。
- 后续若要做 try-out，应采用显式 opt-in，并由项目提供 upstream、auth、frame codec 扩展，不在默认 docs UI 中猜测。
