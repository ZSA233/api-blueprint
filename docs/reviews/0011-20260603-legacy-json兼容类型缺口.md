# legacy JSON 兼容类型缺口

## 状态

部分修复，继续跟踪。

## 发现日期

2026-06-03

## 背景

真实旧接口中存在字段 JSON shape 漂移：同一个字段可能在不同服务实现、历史版本或异常路径中返回 string、integer number、array 或更复杂的组合。客户端生成物如果只按单一 `String` / `Int` / `Array` 解码，Swift、Kotlin、Java、Python 等严格 codec 会直接失败；如果长期退化为 `JSONValue`，业务层又会丢失可读类型和生成器帮助。

## 风险分级

高。问题会影响协议契约可接入性、跨端一致性和生成 SDK 可用性，尤其是移动端发布后难以及时修复的路径。

## 问题性质

兼容性缺口与协议表达缺口。它不是推荐的新接口建模方式，而是旧接口迁入 api-blueprint 时需要的受限逃生通道。

## 存在性判断

已确认。典型复现场景是服务端返回：

```json
{
  "uid": 1000010,
  "roomId": 100
}
```

而客户端 DTO 声明为 string 字段；Swift `Codable` 等严格解码会报类型不匹配。

## 复现场景

- 字段声明为 `String`，wire 返回 integer JSON number。
- 字段声明为 `Array[String]`，wire 偶发返回单个 string。
- 字段声明为 `Array[String]`，array 内部元素混合 string / integer number。

## 影响范围

- Blueprint DSL 无法精确表达旧字段有限多 shape。
- ContractGraph manifest 无法让 writer 区分“任意 JSON”与“有限 legacy union”。
- Swift / Kotlin / Java / Python 等严格解码 target 可能直接失败。
- TypeScript 可用 union 表达，但需要 ContractGraph 提供结构化输入。
- gRPC/protobuf 需要使用 protobuf 原生 `oneof` 或退回 `JSONValue`，不能直接继承 JSON union。

## 兼容性 / 修复风险

新增 DSL 类型本身是向后兼容的；普通字段行为不应变化。风险主要在生成器落地差异：如果某些 target 过早生成过强 wrapper，可能造成 API surface 扩张或编译风险；如果只退到宽泛 JSON 承载，又会降低类型收益。`CoerceString` 还必须拒绝 bool、小数、object 和 array，避免把服务端问题静默吞掉。

## 是否建议修复

建议修复，并按阶段推进：

- 首阶段提供 `OneOf(...)`、`CoerceString(...)`、`LegacyStringID`、ContractGraph manifest、gRPC unsupported 检查和主要 client target 的最小生成支持。
- 后续阶段继续把 Go / Flutter / Kotlin / Java 的 `OneOf` 从 JSON/any/JsonNode 承载收紧到语言原生 wrapper / sealed 类型，并补充端到端 conformance。

## 后续处置建议

- 保持 `OneOf` 文档定位为 legacy compatibility，避免新接口滥用。
- 继续补齐各 target 的 decode helper 行为一致性，特别是声明顺序匹配和严格 scalar 判断。
- 在 examples / conformance 中加入有限 legacy 字段场景，覆盖 `LegacyStringID`、`OneOf(String, Array[String])` 和 `Array[OneOf(String, Int)]`。
- 当所有主要 target 都提供语言原生 wrapper 且 conformance 覆盖后，再将本记录移动到 `docs/reviews/resolved/` 并追加 Resolution。
