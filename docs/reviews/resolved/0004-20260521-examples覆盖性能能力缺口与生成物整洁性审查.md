# Examples 覆盖、性能、能力缺口与生成物整洁性审查

## 状态

已解决并归档。

本记录归档 examples 覆盖、benchmark、能力缺口与生成物整洁性审查的修复闭环。

## 发现日期

2026-05-21

## 背景

完成 media/raw、binary response、生成物 ownership 与客户端 request options 后，examples 已能通过主 conformance 矩阵。但从发布质量角度，还需要确认：

- examples conformance 是否覆盖了新能力的关键边界。
- benchmark 是否仍能反映当前生成物结构，是否存在明显性能退化风险。
- 各端是否还有协议能力缺口。
- 生成物是否因为模板分支调整产生不必要的空行和无意义 diff。

## 风险分级

中高。

主链路已经可用，但以下问题会影响发布前质量判断、性能回归定位、使用者学习成本，以及生成快照的可读性。

## 问题性质

- conformance 覆盖缺口。
- benchmark 失效与性能风险。
- transport 能力边界未完全闭环。
- 生成物格式整洁性与 diff 噪音。

## 存在性判断

存在，且包含一个已确认可复现 bug。

### 1. Java binary benchmark 仍引用旧生成物命名

`scripts/example_benchmark/binary.py` 的 Java benchmark 仍查找：

- `examples/java/client/.../binary/BinaryTypes.java`
- `ApiBinaryBody`

当前 Java 生成物已统一为：

- `GenBinaryTypes.java`
- `GenApiBinaryBody`

复现命令：

```sh
uv run python -m scripts.example_benchmark binary --target java --count 1
```

实际结果：

```text
[Errno 2] No such file or directory: '.../examples/java/client/.../binary/BinaryTypes.java'
```

### 2. conformance 未覆盖 request options 的运行时语义

当前新能力主要通过 codegen 字符串断言覆盖，包括：

- TypeScript / Wails TypeScript `ApiRequestOptions`。
- Java `GenApiRequestOptions`。
- Kotlin / Flutter `ApiRequestOptions`。
- Python keyword-only `headers` / `timeout`。
- Go `runtime.RequestOption`。

但 examples conformance 还缺少跨端真实请求验证：

- default headers 与 per-call headers 的合并、覆盖顺序。
- per-call timeout 覆盖 global timeout。
- timeout 成功触发与正常请求不误触发。
- Wails `timeoutMs` 只限制前端等待、不取消 native call 的边界。

现有 `header` 场景中，部分端仍绕过 generated client 直接发 HTTP 请求，不能证明 request options API 的实际可用性。

### 3. media/raw 边界覆盖仍偏小

当前 `media` 场景覆盖 multipart upload、bytes、file、dynamic filename、MJPEG chunk，但样本都很小，缺少以下边界：

- raw success 不套 JSON envelope 已覆盖，但 raw route typed error / 非 2xx / 空 body / content-type 缺失没有系统覆盖。
- `Content-Disposition` 只覆盖基础 `filename=`，缺少 `filename*=`、非 ASCII 文件名、引号、分号、空文件名与 fallback。
- multipart 大文件、多个 file part、空文件、content type fallback 尚未覆盖。
- byte stream 只读取一个有限 MJPEG 样本，不能验证长流或大流。

### 4. byte_stream 语义跨端仍不完全一致

当前端能力并不等价：

- Go client / Python client / TypeScript client 对 byte stream 更接近真实流式读取。
- Java / Kotlin / Flutter client 当前把 byte stream 缓冲为 `byte[]` / `ByteArray` / `Uint8List`。
- Java / Kotlin server 的 `GenApiStreamResponse` / `ApiStreamResponse` 也是 byte array 模型，不是真持续 streaming writer。

有限响应可用，但如果使用者把 `RSP_BYTE_STREAM` 理解为大文件或无限流，会形成能力误解。

### 5. benchmark 覆盖无法发现生成 client 退化

当前 protocol benchmark 用 `httpx` 直接打 server，只覆盖服务端 HTTP 协议处理，不覆盖各语言 generated client 的热路径。

binary benchmark 覆盖 Go / TypeScript / Python / Kotlin / Java 的 binary encode 路径，但：

- Java benchmark 已因旧命名失效。
- 各端 benchmark 工作负载不完全等价。
- 只有 Go 支持 `--compare-head`，其他端没有直接回归对比。
- 未覆盖 request options、multipart、raw response、file、byte stream 等新路径的性能。

### 6. 生成物存在不必要空行与 diff 噪音

TypeScript `gen_client.ts` 在 request 类型块中会生成多余空行，例如：

```ts
async default(
  request: {
    query?: Types.DefaultQuery;

  } = {},
  options?: ApiRequestOptions,
)
```

无 request data 的 route 或只含少量字段的 route，也容易出现注释块或 request object 内的空行。该问题不影响编译，但会造成：

- examples refresh 出现大量无意义 diff。
- 用户阅读生成物时误以为模板分支缺失内容。
- 后续 review 难以区分真实语义变更与格式噪音。

需要进一步排查其他端是否存在类似模式，尤其是 Java/Kotlin/Flutter/Python 生成物中的空行、尾随空白、空 object / 空参数块。

## 复现场景

- 运行 Java binary benchmark。
- 打开 `examples/typescript/api/routes/api/conflict/gen_client.ts`，观察 `request` 类型块中的空行。
- 打开 `src/api_blueprint/writer/templates/typescript/gen_client.ts.j2`，可见多个可选字段分支之间直接换行，字段不存在时留下空行。
- 运行 conformance `header` 场景，部分端未通过 generated client request options 发起请求。

## 影响范围

- `scripts/example_benchmark/*`
- `scripts/example_conformance/*`
- Java / Kotlin / Flutter / TypeScript / Python / Go generated client 快照。
- HTTP raw media、binary response、request options 的发布质量判断。
- Wails/gRPC transport-native equivalent 文档与示例可信度。

## 兼容性 / 修复风险

- 修复 benchmark 旧命名引用是低风险，应与 Gen 命名规则一致。
- 增加 conformance 场景会提升验证成本，尤其 timeout 场景需要避免 flaky；应使用短超时、专用慢 route 或可控 fake transport。
- 真正统一 byte stream 为跨端流式 API 可能是破坏性改动；如果短期不统一，应先明确文档语义和 capability matrix。
- 清理生成物空行通常不改变 public API，但会刷新大量 snapshots；应单独提交，避免和语义改动混在一起。

## 是否建议修复

建议修复，但建议拆分执行：

1. 先修复 Java binary benchmark，使基准测试恢复可运行。
2. 增加 request options 的运行级 conformance 或 example tests。
3. 明确 `byte_stream` 的跨端语义：真流式补齐，或正式标注部分端为 buffered response。
4. 增加 media/raw filename 与错误路径边界测试。
5. 单独做生成物空行清理与 snapshot refresh。

## 后续处置建议

- 新增 benchmark ownership / smoke 测试，至少保证 `scripts.example_benchmark binary --target java --count 1` 不再因旧路径失败。
- protocol benchmark 增加 generated client 模式，或新增轻量 SDK benchmark，覆盖 request options、binary response、multipart、raw response。
- conformance 新增 `request-options` 场景，使用所有支持 HTTP client 的端真实传递 header，并验证 per-call timeout。
- media conformance 新增 `media-filename-edge` 与 `media-error` 场景。
- 对 `gen_client.ts.j2` 先做模板空白控制；随后扫描 Java/Kotlin/Flutter/Python 生成物，建立“最大连续空行”或“空 request block”测试。
- 如决定暂不补齐 Java/Kotlin/Flutter 真流式 byte stream，应在 docs 和 capability matrix 中写明 buffered 限制。

## Resolution

修复日期：2026-05-21

修复摘要：

- Java binary benchmark 改为使用 `GenBinaryTypes`、`GenApiBinaryBody` 与 `Gen*` runtime 文件，并增加 Java/all benchmark smoke 测试。
- 新增 `sdk-smoke` benchmark 入口，复用 conformance runner 覆盖 generated client SDK 的 request-options、binary-response、media 最小热路径。
- 新增 `request-options` conformance 场景，真实验证 default/per-call header 合并覆盖、global/per-call timeout 覆盖和短 timeout 失败路径。
- 新增 `media-filename-edge` 与 `media-error` conformance 场景，覆盖 `filename*=` 非 ASCII 文件名解析和 raw route typed error envelope 识别。
- Java/Kotlin/Flutter `byte_stream` 改为真流式 API；Go/Python/TypeScript 保持现有真流式路径，并通过互通场景覆盖读取/关闭行为。
- 清理 TypeScript request block 空白控制，增加生成物格式 invariant；Flutter 生成源码也纳入尾部空白归一化，避免 EOF 空白行和无意义 diff。
- 更新 examples snapshots 与 PRE_README、README、configuration、generators、examples-validation、benchmarks 文档。
