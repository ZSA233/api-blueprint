# Client / Server 生成物逃生通道与可用性审查

## 状态

已审查并归档。结论：未发现因为 `gen_*` / `Gen*` 生成物不可手动修改而导致 client 或 server 端在真实使用场景中完全无法达成目的的致命能力缺口。

本记录放入 `docs/reviews/resolved/`，原因是本次审查没有确认必须立即修复的阻塞性问题。文中列出的残余点属于易用性或默认策略优化建议，不构成当前设计不可用。

## 发现日期

2026-05-21

## 背景

前序 review 已经完成生成物 ownership、request options、media/raw、binary response、真流式 `byte_stream` 等补齐。新的问题是：从使用者角度看，`Gen*` / `gen_*` 文件不可修改后，面对长耗时请求、网络环境差异、并发、资源占用、cookie、请求自定义、上下文管理、server 端框架策略等场景，是否会出现“生成物不支持，用户又不能改生成物，所以只能用很丑或不可维护方式绕过”的致命缺口。

## 风险分级

低。

审查发现各端都至少具备以下一种逃生通道：

- client 端可注入自定义 transport、原生 HTTP client、fetcher 或请求 options。
- server 端可注入 service 实现、挂载框架 middleware/plugin/filter/provider，或在 preserved scaffold/facade 中包装生成物。
- Wails / gRPC 不试图复用 HTTP 语义，而是走各自协议原生的 deadline、metadata、IPC envelope、service implementation 等扩展路径。

## 问题性质

架构可用性审查，不是已确认 bug 修复记录。

重点审查以下问题：

- client 是否能设置 per-call timeout / deadline、headers、cookie、credentials、代理、TLS、连接池、重试、取消、并发控制。
- client 是否能处理大文件、真流式响应、长连接资源关闭。
- server 是否能插入鉴权、限流、日志、trace、上下文、取消、超时、文件大小限制、框架级安全策略。
- 当默认生成适配器不满足项目要求时，用户是否有稳定的非生成文件或接口可以替换，而不需要编辑 `gen_*` / `Gen*`。

## 存在性判断

未确认致命能力缺口。

各端的默认生成物不一定覆盖所有生产策略，但都提供了直接能力或可维护逃生通道。当前设计的核心自洽点是：生成物负责协议契约和默认适配器；宿主应用通过 preserved facade、service implementation、transport interface、框架 middleware/plugin、原生 client 注入来承担环境策略。

## Client 端结论

| Target | 直接能力 | 逃生通道 | 结论 |
| --- | --- | --- | --- |
| Go HTTP | 路由方法接收 `context.Context` 与 `runtime.RequestOption`；HTTP config 可注入 `*http.Client`、default headers；`byte_stream` 返回 `io.ReadCloser`。 | 自定义 `runtime.Transport`；preserved route/client 文件可包一层；原生 `http.Client` 可配置 timeout、cookie jar、proxy、TLS、连接池、重试 wrapper。 | 无致命缺口。 |
| TypeScript HTTP | `ApiRequestOptions` 支持 headers、`RequestInit`、`timeoutMs`；HTTP config 支持 `fetcher`、default headers、global init、timeout；stream 返回原生 body stream。 | 自定义 `ApiTransport` 或 `fetcher`；facade 可导出用户包装 client。 | 无致命缺口。cookie/credentials 通过 `init.credentials` 或自定义 fetcher 处理。 |
| TypeScript Wails | 复用 request options；headers 进入 envelope；`timeoutMs` 只控制前端等待，不承诺取消 native call。 | 自定义 Wails transport；Go service implementation 侧处理真实取消或长任务策略。 | 无致命缺口，语义边界清楚。 |
| Flutter HTTP | `ApiRequestOptions` 支持 headers、timeout；HTTP config 支持 default headers、global timeout、注入 `http.Client`；`byte_stream` 返回 `Stream<List<int>>`。 | 自定义 `ApiTransport`；注入带 cookie/retry/proxy 行为的 `http.Client`；preserved `http_api_client.dart` 可包装。 | 无致命缺口。 |
| Kotlin HTTP | `ApiRequestOptions` 支持 headers、`Duration` timeout；config 支持 suspend default headers、global timeout；可注入 `OkHttpClient`；`byte_stream` 是 closeable `InputStream` wrapper。 | OkHttpClient 原生拦截器、cookieJar、proxy、dispatcher、connection pool、TLS；自定义 `ApiTransport`。 | 无致命缺口。 |
| Java HTTP | `GenApiRequestOptions` 与 route overload 支持 per-call headers/timeout；config 支持 default headers/global timeout；JDK `HttpClient` 和 `ObjectMapper` 可注入；`byte_stream` 使用 `InputStream`。 | preserved `ApiClient` 可接收任意 `GenApiTransport`；自定义 `HttpClient` 处理 cookie、proxy、TLS、executor、HTTP version。 | 无致命缺口。 |
| Python HTTP | route 方法暴露 keyword-only `headers`、`timeout`；transport 可注入 `httpx.AsyncClient`；stream 响应提供 `aclose()`。 | `httpx.AsyncClient` 原生支持 timeout policy、cookies、proxy、limits、event hooks；可实现 `ApiClientTransport`。 | 无致命缺口。 |
| gRPC | api-blueprint 不生成额外 client wrapper。 | 使用 native gRPC deadline、metadata、interceptor、call options、channel config。 | 非缺口，属于协议原生边界。 |

### Client 场景判断

- 长耗时请求：Go 使用 `context.Context`，Java/Kotlin/Flutter/TypeScript/Python 均有 per-call timeout 或原生 client timeout；不希望超时的场景可以不给 per-call timeout，并使用原生连接配置。
- 网络环境与并发：Go `http.Client`、Java `HttpClient`、Kotlin `OkHttpClient`、Python `httpx.AsyncClient`、Flutter `http.Client`、TypeScript `fetcher` 都可注入或替换。
- cookie / credentials：TypeScript 通过 `RequestInit`，Go/Java/Kotlin/Python/Flutter 通过原生 client 或 transport wrapper。
- 大响应和资源释放：`byte_stream` 在 Go/Kotlin/Java/Python/Flutter/TypeScript 都有流式或 closeable/readable 形态；用户需要按语言习惯关闭 `ReadCloser`、`InputStream`、`httpx` stream 或取消订阅。
- 特殊请求自定义：默认适配器不直接覆盖所有参数，但可通过 `ApiTransport` / `GenApiTransport` / `runtime.Transport` / `httpx.AsyncClient` / `fetcher` 等稳定接口替换。

## Server 端结论

| Target | 直接能力 | 逃生通道 | 结论 |
| --- | --- | --- | --- |
| Go HTTP/Wails | service/router implementation 为 preserved；provider pipeline 暴露 route info、headers、metadata、base context；HTTP 侧可拿 Gin context 能力。 | Gin middleware、custom provider、custom route mount、自定义 transport adapter；Wails service implementation。 | 无致命缺口。 |
| Java Spring | service interface/generated controller 分离；用户实现 `DemoService` bean；ObjectMapper 与 Spring MVC 基础设施可由应用配置。 | Spring filter/interceptor/security/configuration；需要深度控制时可不使用默认 controller 或提供 app-owned controller。 | 无致命缺口。存在默认策略偏硬的优化点。 |
| Kotlin Ktor | `Route.register*Routes(service = ...)` 允许注入 service；用户可在 Ktor route scope 外层挂 auth、rate limit、logging、timeout、content negotiation；service 为 suspend，协程取消可传播。 | Ktor plugin、route nesting、自定义注册顺序、app-owned wrapper route。 | 无致命缺口。 |
| Python FastAPI | `create_router(...service=...)` 接收 service implementation；router 可由 app include，并套 middleware、dependencies、lifespan、exception handler。 | FastAPI middleware/dependencies/contextvars；必要时自定义 router wrapper 或实现替代 adapter。 | 无致命缺口。service 方法未直接接收 `Request` 是易用性优化点。 |
| Wails | Go service implementation 是用户落点；TypeScript transport 只是 IPC wrapper。 | Wails app/service 生命周期、context、权限、文件系统策略由宿主实现。 | 无致命缺口。 |
| gRPC | proto 与 protoc 产物保留原生 server interface。 | native gRPC server interceptor、context、metadata、deadline、stream API。 | 非缺口，属于协议原生边界。 |

### Server 场景判断

- 鉴权、trace、日志、限流：Go 走 provider/Gin middleware，Java 走 Spring filter/interceptor/security，Kotlin 走 Ktor plugin/route nesting，Python 走 FastAPI middleware/dependencies。
- 上下文和取消：Go provider context 包装 `context.Context`；Kotlin suspend route 可随 coroutine 取消；Python/FastAPI 与 Java/Spring 可通过框架 request lifecycle 和服务实现处理。
- 文件大小、上传策略、资源占用：默认 adapter 有基础实现；生产环境可通过框架配置或自定义 adapter 调整。Go 的 multipart 内存阈值目前在 generated HTTP adapter 中固定为 `32 << 20`，但可通过 Gin/adapter 路径绕过，不构成完全不可用。
- 长连接：Go/Kotlin/Python/TypeScript/Flutter client 侧已经有 stream/channel bridge；Java Spring 默认 controller 能跑 SSE/WebSocket，但默认超时和 origin 策略偏硬。

## 非致命优化建议

这些点不阻塞当前设计，但后续可以作为单独优化计划处理：

- Java Spring server 的 `SseEmitter(30_000L)` 和 WebSocket `setAllowedOrigins("*")` 是 generated 默认值，生产项目更希望有配置入口或更明确的文档说明。
- Python server service 方法目前不直接接收 `Request` / typed request context。用户可用 middleware、dependencies、contextvars 或 router wrapper 解决，但一等 context 参数会更顺手。
- Go HTTP server multipart `ParseMultipartForm(32 << 20)` 默认阈值写在 generated adapter 中，复杂上传场景更希望可配置。
- TypeScript / Flutter 默认 WebSocket adapter 暴露的是常用能力；更细粒度握手、代理、证书、重连策略需要自定义 transport 或底层 client。
- 各端文档可以进一步把“默认 adapter 能力”和“生产逃生通道”分开列示，降低用户误以为必须修改 `Gen*` 文件的学习成本。

## 示例判断

以下使用方式都不需要编辑生成文件：

### Java client 注入原生 HTTP client

```java
var client = new ApiClient(
    new GenJdkHttpApiTransport(config, customHttpClient, customObjectMapper)
);
```

### TypeScript client 使用 credentials 与 timeout

```ts
const clients = createClients({
  baseUrl,
  init: { credentials: "include" },
  timeoutMs: 3000,
});
```

### Go client 使用 context 与 per-call header

```go
ctx, cancel := context.WithTimeout(ctx, 3*time.Second)
defer cancel()

result, err := api.Demo.RequestOptions(ctx, query, runtime.WithHeader("x-trace-id", traceID))
```

### Python client 注入 httpx.AsyncClient

```python
transport = HttpClientTransport(
    base_url,
    client=httpx.AsyncClient(timeout=timeout, cookies=cookies, limits=limits),
)
client = ApiClient(transport)
```

### Kotlin client 注入 OkHttpClient

```kotlin
val api = createHttpApiClient(config, okHttpClient)
```

### Flutter client 注入 transport

```dart
final api = createHttpApiClient(
  config: config,
  transport: customTransport,
);
```

## 影响范围

本审查影响的是用户对生成 SDK / server adapter 的使用判断，不改变 DSL、writer、examples 或 runtime 行为。

当前不需要刷新 examples，也不需要新增 conformance 场景。后续如果把非致命优化点升级为功能计划，应单独补测试。

## 兼容性 / 修复风险

当前不建议立即修改公共 API。

若后续实现更细的 server context/config 能力，需要注意：

- Java Spring controller 配置化可能改变默认 SSE/WebSocket 行为，应保持默认兼容或在 migration notes 中说明。
- Python server 增加 request context 参数可能影响 service interface，是潜在 breaking change，应考虑新增可选 context 或 wrapper，而不是直接破坏现有 service 方法。
- Go multipart 阈值配置化应优先走 config/provider，不应把宿主运行时策略扩大到 DSL 语义。

## 是否建议修复

不建议作为阻塞性修复处理。

建议把非致命优化点作为后续文档或小型配置增强处理，优先级低于协议正确性、conformance 覆盖和真实编译/运行失败。

## 后续处置建议

- 在 generator / configuration 文档中增加“生产逃生通道”小节，明确用户不应修改 `Gen*` / `gen_*`，而应替换 transport、注入原生 client、使用 service implementation 或框架 middleware。
- 如果后续收到真实生产场景反馈，再把 Java Spring SSE/WebSocket 默认值、Python service context、Go multipart 阈值拆成独立 plan。
- 保持 conformance 聚焦协议行为；环境策略类能力通过 codegen 单测和文档示例证明即可。
