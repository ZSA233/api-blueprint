# Client / Server 安全并发裁剪与依赖风险审查

## 状态

已修复并归档。

本次审查未发现必须立即停止使用的阻塞风险；确认的中风险与低风险问题已通过安全默认、配置入口、file part 流式表示、窄入口文档和依赖审计可重复性补齐完成闭环。因此记录移动到 `docs/reviews/resolved/`。

## 发现日期

2026-05-22

## 背景

前序审查已经确认 client/server 生成物存在可维护逃生通道，不会因为 `gen_*` / `Gen*` 不可手改而完全不可用。本次审查进一步从攻击者、生产高并发、SDK 使用者和依赖维护者四个角度检查：

- 安全性：攻击者是否可以通过大请求体、gzip bomb、multipart、WebSocket、SSE、raw/file response、错误回显等路径扩大攻击面。
- 高并发与资源占用：默认生成 adapter 是否容易出现无界队列、线程池耗尽、内存暴涨、流未关闭、长连接堆积。
- 未引用安全裁剪：只使用部分 route/root/transport 时，是否仍强制拖入大量未使用生成物或依赖。
- 依赖过时 / 冲突 / 锁死：生成物是否硬锁依赖，示例工程是否可审计，是否存在明显过时或 alpha 依赖。

中。

## 存在性判断

审查时确认存在需要追踪的问题；截至 2026-05-22，已通过本记录的 Resolution 完成修复闭环。

这些问题多数可以通过宿主框架、反向代理、自定义 transport/service 或 preserved facade 兜底，但默认生成物目前没有把安全边界显式建模，容易让使用者误以为默认 adapter 已经适合公网生产部署。

## 中风险问题

### 1. 默认 server adapter 缺少统一请求体上限

攻击视角：攻击者可构造超大 JSON、binary body、multipart file 或压缩后膨胀的数据，迫使默认 adapter 在进入业务逻辑前消耗大量内存或 CPU。

确认点：

- Go HTTP adapter 使用 `ParseMultipartForm(32 << 20)`，但没有统一 `MaxBytesReader` 或 route-level body limit；binary gzip 通过 `gzip.NewReader` 后直接交给 decoder，缺少解压后字节上限。
- Java Spring controller 对 multipart file 调用 `MultipartFile.getBytes()`，会把文件读入内存后包装成 `GenApiFilePart`。
- Kotlin Ktor `decodeMultipart` 对 file part 使用 `readRemaining().readBytes()`，并进一步把 bytes 展开成 `JsonArray`。
- Python FastAPI binary route 使用 `await request.body()`，multipart 使用 `await request.form()`；默认 adapter 没有生成 route-level 大小限制。

影响：

- 高并发或恶意输入下，server 可能在业务鉴权前发生内存放大。
- 宿主框架和反向代理可以兜底，但生成物没有暴露统一配置入口，学习成本高。

建议：

- 为 HTTP server adapter 增加可配置的 request body / multipart / binary decode 上限。
- 默认文档明确：公网部署必须在 reverse proxy 或框架层设置 request size、multipart、decompression limit。
- 对 file part 尽量保留 stream/file descriptor 形态，避免默认转成内存 bytes。

### 2. WebSocket / SSE 默认长连接资源边界偏弱

攻击视角：攻击者建立大量连接、持续发送消息或让业务 receive 阻塞，消耗线程池、队列和连接资源。

确认点：

- Java Spring WebSocket 默认 `setAllowedOrigins("*")`，没有生成配置入口。
- Java Spring channel 使用无界 `LinkedBlockingQueue`，持续入站消息可导致内存增长。
- Java Spring SSE / WebSocket handler 使用 `CompletableFuture.runAsync()` 默认 common pool，连接数上升时可能影响应用内其他异步任务。
- Java Spring SSE 使用固定 `SseEmitter(30_000L)`，无法按 route 或应用配置调整。
- Python `_SseStream` 使用无界 `asyncio.Queue`。
- Go HTTP WebSocket 使用默认 accept，当前没有生成 origin policy、message size limit 或 compression 策略配置。

影响：

- 不一定导致功能错误，但默认公网暴露时抗压能力不足。
- 这类风险很容易被误认为“框架默认会处理”，实际上生成 adapter 已经选择了具体队列、origin、executor、timeout 策略。

建议：

- Java server 增加可配置 allowed origins、SSE timeout、executor、inbound queue size。
- Python SSE queue 增加上限或 backpressure 策略。
- Go WebSocket 增加 origin policy、message size / compression 选项。
- 文档声明默认长连接 adapter 是开发友好默认值，不是生产安全策略。

### 3. Multipart / raw file 语义混合了协议契约和内存实现

攻击视角：攻击者可利用“协议字段是文件”但默认实现转为 bytes 的事实，构造大文件触发内存峰值。

确认点：

- Java/Kotlin/Flutter runtime 中 `ApiFilePart` 主要以 bytes/list 形态表达。
- Java/Kotlin server multipart adapter 把上传文件读入内存。
- Python multipart adapter 依赖 Starlette form parsing，未生成文件大小和 spool 策略说明。

影响：

- 小文件和测试场景可用，但生产文件上传默认不够稳。
- 这不是 wire contract 错误，而是 runtime representation 的资源风险。

建议：

- 后续 review/plan 中把 file part runtime type 分成 small bytes 与 stream/file descriptor 两类。
- 对当前 bytes 形态明确标注适用范围和上限建议。

## 低风险问题

### 1. 未引用裁剪依赖于用户选择窄入口

确认点：

- TypeScript `api/transports/gen_clients.ts` 同时导入 HTTP、Wails v2、Wails v3 factory；用户只想用 HTTP 但从总入口导入 transports 时，会把 Wails factory 纳入模块图。
- TypeScript root `gen_index.ts` 导出 Runtime / Routes / Transports，适合发现能力，但不是最窄导入路径。
- Flutter `lib/api.dart` 导出整个 root `gen_api.dart`；`GenApiClient` 聚合所有 route clients。
- Python server `create_router()` 在单个 root router 内导入并注册该 root 下所有 services/types；只想部署部分 route 时没有生成的 per-group router 入口。
- Java/Kotlin suite 示例把 client/server 生成树整体纳入编译；这是示例工程行为，不一定代表真实用户工程。

影响：

- 安全上不是直接漏洞，但会增加 bundle、编译面、依赖扫描面和攻击面理解成本。
- 对浏览器/桌面客户端，未使用 transport 被 barrel 导入时可能影响 tree-shaking 效果。

建议：

- 文档推荐生产项目使用窄入口，例如只导入具体 route client、具体 HTTP factory，不从总 `transports` barrel 进入。
- 后续生成 per-group / per-transport 的更窄 facade，避免只用 HTTP 时导入 Wails factory。
- Python server 增加可选 per-group router factory，允许只 include 实际部署的 routes。

### 2. 依赖审计可重复性不足

确认点：

- `examples/wails-harness/v2/frontend` 和 `examples/wails-harness/v3/frontend` 没有 `package-lock.json`，`npm audit --omit=dev` 直接失败，错误为 `ENOLOCK`。
- `examples/wails-hello/app/frontend` 有 lockfile，`npm audit --omit=dev` 返回 0 个生产依赖漏洞。
- Flutter example 没有 `pubspec.lock`，`dart pub outdated` 明确提示 `No pubspec.lock found`，只能看到可解析版本，不能看到当前锁定版本。
- Python `uv.lock` 存在，`uv tree` 可解析当前依赖；`pip-audit` 在本机工具环境失败，未形成漏洞结论。

影响：

- 没有 lockfile 的示例无法稳定复现依赖审计结果。
- 这主要影响 example validation / harness 的发布质量，不是生成代码本身的运行时契约缺陷。

建议：

- 为需要审计的 Node / Flutter examples 增加 lockfile，或在文档中明确这些 harness 不是可复现安全基线。
- CI 增加依赖 audit smoke，至少覆盖有 lockfile 的示例。

### 3. 示例依赖存在过时或 alpha 依赖

确认点：

- Go server example：`gin v1.10.0 -> v1.12.0`、`coder/websocket v1.8.13 -> v1.8.14`，多个 `golang.org/x/*` 间接依赖明显可升级。
- gRPC Go example：`grpc v1.81.0 -> v1.81.1`，多个 `golang.org/x/*` 和 `google.golang.org/genproto` 可升级。
- Wails v3 examples 使用 `github.com/wailsapp/wails/v3 v3.0.0-alpha.74`，frontend `@wailsio/runtime` package 最新为 `3.0.0-alpha.79`。
- Wails hello frontend：`npm audit --omit=dev` 无生产漏洞；`vite 5.4.21 -> 8.0.14`、`typescript 5.9.3 -> 6.0.3` 属于 dev dependency major 可升级。
- Flutter example：`http` 可解析到 `1.6.0`，`web_socket_channel` 可解析到 `3.0.3`；dev dependency `lints` 最新为 `6.1.0`。
- Java suite：Gradle runtimeClasspath 显示 Spring Boot `3.3.6`，Tomcat `10.1.33`；Maven metadata 当前可见 `spring-boot-starter-web` 最新 release tag 为 `4.1.0-RC1`，`jackson-databind` 最新 `2.21.3`，`tomcat-embed-core` 最新 `11.0.22`。这些是示例工程依赖，不是生成物硬锁。

影响：

- 示例工程不是用户生产依赖真源，但示例被当作参考时，会传播旧版本或 alpha 依赖。
- Wails v3 alpha 是明确稳定性风险，应在文档中继续标记为 experimental。

建议：

- 单独做 examples dependency refresh 计划，避免把依赖升级和生成器语义修复混在一起。
- Wails v3 继续标记 experimental；不要把 alpha runtime 当作稳定目标宣传。

| Target | 安全与资源边界 | 高并发风险 | 依赖锁死判断 |
| --- | --- | --- | --- |
| Go HTTP/Wails | provider/Gin context 有逃生通道，但默认 HTTP adapter 没有统一 request body limit；WebSocket accept 缺 origin/message-size 默认配置。 | multipart 阈值固定；gzip 解码无生成上限；长连接依赖业务正确关闭。 | Go server example 依赖较旧，但用户可升级 module。 |

| Python FastAPI | FastAPI middleware/dependencies 可兜底；默认 adapter 对 body/form 没有生成上限。 | binary request 全量 `await request.body()`；SSE queue 无界。 | Python package 有版本范围和 lock；示例 service 不硬锁用户 app。 |

| gRPC | 使用 native gRPC metadata/deadline/interceptor，api-blueprint 不生成 server runtime 策略。 | streaming backpressure 交给 gRPC runtime。 | protoc/gRPC 依赖由用户生态控制；example 可升级补丁版本。 |

## 复现场景 / 攻击视角

- 对 multipart route 连续上传大文件，Java/Kotlin 默认 adapter 会把 file part 转成内存 bytes。
- 对 Python binary route 发送超大 body，默认 route 在解析前 `await request.body()`。
- 对 Go binary route 发送 gzip 压缩后膨胀体，默认 adapter 没有生成级解压后大小限制。
- 对 Java WebSocket route 快速发送消息但业务 `receive()` 慢处理，无界 `LinkedBlockingQueue` 会持续增长。
- 对 Java WebSocket route 从任意 origin 建连，默认 `setAllowedOrigins("*")` 不拦截。
- 项目只想用 TypeScript HTTP client，但从 `Transports` 总 barrel 导入时会把 Wails factory 纳入模块图。
- 在 Wails harness frontend 执行 `npm audit --omit=dev`，无 lockfile 时无法审计。

## 影响范围

- 影响默认 HTTP server adapter 的生产安全基线。
- 影响 examples 作为长期参考时的依赖健康与可审计性。
- 不影响 DSL wire contract 的正确性。
- 不影响已经确认的 client/server 逃生通道存在性；宿主仍可通过框架配置、反向代理、自定义 transport/service、窄入口导入等方式兜底。

## 兼容性 / 修复风险

- 增加默认 request body 上限可能改变现有大请求行为，需要可配置并提供迁移说明。
- Java WebSocket queue 从无界改为有界会引入 backpressure/drop/close 策略选择，必须明确默认行为。
- Multipart file part 从 bytes 改为 stream/file descriptor 可能影响各端公开类型，属于潜在 breaking change。
- 裁剪入口调整应保留现有总 barrel，但新增窄入口或文档推荐，避免破坏现有 import。
- 依赖升级应独立于生成器行为修复，避免把依赖 churn 混入协议变更。

## 是否建议修复

已修复并归档。

原建议修复优先级：

1. 先修 server 资源边界：body/multipart/binary 上限、Java WebSocket queue/executor/origin/SSE timeout、Python SSE queue。
2. 再补文档：默认 adapter 的生产安全边界、反向代理/框架兜底示例、窄入口导入建议。
3. 再做 examples 依赖刷新与 lockfile 策略。
4. 最后评估 file part stream/file descriptor 的 breaking change。

## 原后续处置建议

- 新增计划文档，拆分为 “server adapter resource guard” 和 “examples dependency/cut-through cleanup” 两组。
- 对中风险项新增 codegen 测试或静态 invariant：
  - Java WebSocket queue 不允许无界默认。
  - Java WebSocket allowed origins 必须可配置。
  - HTTP server generated config 必须包含 body/multipart limits。
  - Python/Java/Kotlin multipart 不应无条件把文件读入无上限 bytes。
- 对低风险项新增文档：
  - 推荐生产使用窄入口 import。
- Wails v3 alpha 仍为 experimental。
- examples harness 没有 lockfile 时不作为安全审计基线。

## Resolution

修复日期：2026-05-22

关联计划：

- `docs/plans/resolved/0004-20260522-production-escape-hatches-and-narrow-entrypoints.md`
- `docs/plans/resolved/0005-20260522-server-resource-guards-file-streaming-and-dependency-hygiene.md`

修复摘要：

- Go HTTP server 新增生成的 `ServerConfig` 与默认安全上限：request body `16 MiB`、multipart memory `8 MiB`、single file `32 MiB`、decompressed binary `16 MiB`；WebSocket 默认保留 origin 校验、禁用 compression，可显式配置 origin pattern 或跳过校验。
- Java Spring server 新增 `GenSpringServerConfig`，默认不再 `setAllowedOrigins("*")`；SSE timeout、WebSocket inbound queue capacity、executor、multipart single file limit 均可配置，超大 multipart 返回 413。
- Kotlin Ktor server 新增 `ApiServerConfig`，对 multipart file、binary body、WebSocket message 启用默认上限，并通过 route registration 参数暴露覆盖入口。
- Python FastAPI server 新增 `ApiServerConfig`，对 body、multipart file/part、SSE queue、WebSocket message 启用默认上限；新增 per-group router factory，aggregate `create_router()` 保持兼容。
- Java/Kotlin/Flutter file part runtime 迁移为 stream/file descriptor 优先，保留 `fromBytes` / `readAllBytes()` 便利 helper；Python multipart client/server 支持 file-like / UploadFile-compatible 输入，并按配置上限拒绝超大上传。
- Python HTTP client multipart 改为 async-safe multipart encoder，避免 `httpx.AsyncClient` 直接接收同步 `files=` stream 时触发 “sync request with AsyncClient”。
- 文档已补充默认 adapter 的生产边界、宿主 middleware/plugin/filter/native client 逃生通道、窄入口导入建议、Wails timeout 语义、gRPC native deadline/metadata/interceptor 边界。
- 示例依赖做保守卫生处理：Wails frontend 与 Flutter example 提交 lockfile；Wails runtime、Go server/gRPC/Wails Go 依赖做 patch/minor 刷新，不跨 Spring/Vite/TypeScript major。

兼容性记录：

- server 安全默认会拒绝超过默认上限的请求；生产项目可通过新增 config 类型显式放宽。
- file part API 允许 stream/file descriptor 路径，这是面向大文件的 breaking cleanup；小文件测试路径仍可使用 bytes helper。
- 旧 aggregate client/server/barrel 入口保留；新增或推荐窄入口不破坏现有 import。

依赖审计记录：

- `npm audit --omit=dev --json` 已覆盖 Wails frontend lockfile，生产依赖漏洞为 0；普通 npm install 仍提示 Vite/esbuild 链上的 dev-only moderate 风险，本轮不跨 major 处理。
