# api-blueprint AI Agent Guide

## 目的

本文件约束 AI agent 在 `api-blueprint/` 仓库中的开发、调试、测试与发布准备方式。

目标不是只把功能改出来，而是让改动同时具备以下特征：

- 可验证
- 可回退
- 低耦合
- 方便后续 AI 和人类继续维护

若某个场景未被单独覆盖，默认遵循：

- 优先最小必要改动
- 优先保留证据而不是堆补丁
- 优先清晰结构而不是隐式约定

## 基本原则

- 修复问题时，先区分“猜测性修复”和“已确认修复”。
- 任何影响公共行为的改动，都要同时补测试或补可验证的检查路径。
- 不要为了可能的未来需求引入长期复杂度。
- 临时排查代码在问题定位后必须明确保留还是删除。
- 代码组织既要服务人类维护，也要服务大模型上下文理解。

## 仓库职责边界

- `src/api_blueprint/engine/` 是蓝图 DSL、FastAPI/OpenAPI 语义和运行时组装层。
- `src/api_blueprint/writer/` 是代码生成层，负责 Go / TypeScript 输出，不负责 DSL 语义定义。
- `src/api_blueprint/hub/` 与 `src/api_blueprint/static/` 是文档服务展示资源。
- `examples/blueprints/` 是示例蓝图真源。
- `examples/golang/` 与 `examples/typescript/` 是根据示例蓝图生成出的快照产物，不是手写业务源。

修改时必须遵守：

- 不要把 DSL 语义判断塞进 `writer/` 模板绕过 `engine/`。
- 不要手改 `examples/golang/` 或 `examples/typescript/` 的业务逻辑；如果模板、DSL 或示例配置变化导致产物变化，应重新生成并校验。
- 如果只想修复生成产物，先确认真实问题是在模板、writer 还是示例蓝图，而不是直接编辑快照。

## Blueprint 运行时规则

- `Blueprint(app=None)` 默认复用全局 `FastAPI` app；这属于项目既有行为。
- 如果需要拆分多个文档服务或多应用场景，必须显式传入 `app`，不要假设每个 `Blueprint` 默认独立。
- 涉及多蓝图测试时，应显式验证是否共享同一个 app，避免重复注册路由或误判行为。

## 代码耦合与拆分规则

### 1. 文件过大或职责过多时，主动拆分

- 如果一个文件同时承载入口编排、核心语义、配置处理、工具函数、模板适配或测试辅助等多种职责，应主动考虑拆分。
- 如果为了完成一个改动，需要在同一文件里反复跳转多个不相关区域，通常说明这个文件已经过于耦合。
- 不要因为“当前项目一直这样”就继续把历史耦合结构当成规范；现状不是设计真源。

### 2. 拆分按逻辑边界，而不是按当前目录惯性

- 拆分时应围绕稳定的逻辑边界，例如 DSL 语义、生成器实现、文档服务、CLI 装配、配置解析、共享工具或测试辅助。
- 不要机械按行数切块，也不要为了迁就历史目录把本应独立的职责继续绑在一起。
- 如果未来重构需要调整目录层级、模块归属或包结构，只要能显著降低耦合、提升可读性和可验证性，就应优先选择更清晰的边界。

### 3. 文件数量增长后，使用目录或包做组织

- 当相关文件增多时，不要继续平铺堆放。
- 应按功能域使用目录或包进行分组，并通过少量入口文件完成装配与导出。
- 入口文件应尽量保持薄，主要负责组织、调度和对外暴露接口，不承载大量实现细节。

### 4. 测试代码也必须拆分

- 单元测试、集成测试、打包测试、release contract 测试都不应演化成“大而全”的单文件。
- 测试应按模块、行为、场景或错误类型拆分，避免一个测试文件覆盖过多不相关内容。
- 公共 fixture、helper、构造逻辑和断言辅助可以提取到共享测试模块中。

### 5. 结构优化要服务 AI 友好性

- 更短的文件、更单一的职责、更清晰的目录结构，会直接提升 AI agent 的理解效率和修改准确率。
- 当不确定是否应该拆分时，优先选择更容易阅读、搜索、定位、替换和复用的组织方式。
- 拆分后应保持命名清晰、依赖方向稳定、对外接口简单。

## 依赖显式化与入口设计规则

### 1. 入口文件必须薄，且只暴露真实公共入口

- 包根 `__init__`、CLI 根模块、启动入口、对外导出模块主要负责注册、装配和暴露公共接口，不承担内部依赖中转层。
- 内部模块应直接从真实定义位置导入依赖，不要通过包根或入口模块反查内部对象。
- 如果用户没有明确要求内部兼容，不要为了保留旧的内部导入路径、隐式 monkeypatch 入口或历史中转层而继续套壳。

### 2. 类的依赖必须在类自身可见

- 单独打开一个类时，应能从其字段、构造器参数、类型注解和 imports 里看出主要依赖。
- 不允许一个类依赖“宿主对象碰巧提供的属性或方法”才能成立。
- 如果多个组件需要共享运行时依赖，应通过显式参数、context/runtime 对象或清晰字段传入，而不是隐式从 `self` 上取值。

### 3. 默认优先组合，不默认使用隐式 mixin

- 当目标是拆分大类或大文件时，优先使用显式组合结构，例如 `manager/facade + component`、`runtime/context + service`。
- 共享逻辑优先写成纯函数、轻量 helper 或独立组件，不要默认挂在宿主对象私有方法上供其他组件隐式调用。
- mixin 只适用于依赖极少、契约稳定、单看类本身仍然足够清楚的场景；否则不要使用。

### 4. 不要为了内部兼容长期保留魔法中转层

- 公共 API 可以为了兼容性保留，但内部实现形态不默认保留兼容。
- 如果一次重构已经能够明确依赖归属、清理错误导入方向或去掉历史动态壳，就不要因为“以前有人这样 import 过”而继续保留中间层。
- 规范目标优先于历史偶然结构；只要不破坏明确承诺的公共接口，就应允许内部依赖面收缩和重定向。

### 5. 测试必须 patch 显式拥有者

- 测试中的 monkeypatch、stub 和 mock 应命中真实归属模块、真实组件对象和真实拥有者方法。
- 不要把包根、入口模块或碰巧可达的私有 helper 当成长期测试入口。
- 如果重构后依赖拥有者发生变化，应同步重写测试 patch 目标，而不是为了兼容旧测试继续保留错误的内部结构。

## Python 类型注解规则

- Python 属于弱类型语言；如果类型信息长期缺失，后续维护、重构和阅读成本会快速上升。
- 因此在本项目中，AI agent 编写或修改 Python 代码时，应主动补全和维护类型注解，而不是把类型信息留给后续猜测。
- 新增或修改 Python 代码时，优先为函数参数、返回值、类字段、实例属性、配置结构、上下文对象、字典字段等补齐清晰的类型标注。
- 仅仅让代码“能跑”不够；类型信息也应足够清楚，便于后续 AI 和人类理解数据结构与调用约束。
- 如果某个对象结构固定，不要长期使用语义模糊的裸 `dict`、裸 `list` 或大面积 `Any` 应付过去，应优先考虑更明确的表达方式。
- 如果确实无法给出精确类型，也应尽量给出比 `Any` 更窄的近似类型，并在必要时补充注释说明限制原因。
- 对于历史代码，如果本次改动已经触达某段 Python 逻辑，且补类型注解成本合理，应顺手改善局部类型信息。
- 类型注解的目标不是形式化打卡，而是降低理解门槛、减少误用概率、提升重构安全性。
- 注入式依赖必须有明确类型，不要把模块对象、运行时对象、模板环境、文件系统依赖或网络/进程依赖塞进无类型字段。
- 如果需要抽象依赖面，只允许使用小而明确的 `Protocol`，并定义在使用侧；不要用大而泛的接口占位。
- 不要用“返回一个不透明对象或模块，再动态调用其方法”的方式规避类型表达。

## 文档维护规则

- `PRE_README.MD` 是 README 收束的事实基线。
- 只要改动影响用户可见能力、CLI、配置、生成目录、文档服务或发布方式，就必须先更新 `PRE_README.MD`。
- 然后再同步收束到：
  - `README.md`
  - `README_EN.md`
- `README.md` 与 `README_EN.md` 必须严格镜像：
  - 标题层级一致
  - 表格数量和顺序一致
  - 代码块数量和顺序一致
  - 提示项和限制说明一致
- 文档示例、路径示例和配置示例默认使用中性命名；不要写入具体业务仓库、业务域或项目私有名词。
- `main.py` 与 `debug.py` 是本地辅助脚本，不属于对外公共文档主线，不应在 README 中当作正式接口推广。

## 发布规则

- 用户如果只说“发版”，AI agent 不应自行猜目标版本号；必须等用户明确给出版本号。
- 发布工作默认在冻结分支 `release/vX.Y.Z` 上完成，而不是直接在日常开发分支上打 tag。
- 进入正式发版准备前，必须先完成文档收束：
  - `PRE_README.MD`
  - `README.md`
  - `README_EN.md`
- 版本真源只允许通过：
  - `release-version.toml`
  - `src/api_blueprint/_version.py`
- 不要手改版本号散落文件。
- `make release-version-rc` / `make release-version-stable` 只负责切换版本真源与同步检查，不会自动创建 git commit 或 tag。
- release commit、git tag、push tag 必须按顺序显式执行，不要并行，不要把 tag 打到前一个提交上。
- 正式发布前必须先通过：
  - `make release-preflight RELEASE_TAG=...`
  - `make release-local RELEASE_TAG=...`
  - `make release-install-check RELEASE_TAG=...`
- push tag 前必须确认当前 release ref 的 GitHub Actions `CI` 已经全绿。
- stable `Release` workflow 的正式发布 job 必须绑定 GitHub `production` environment。
- 如果仓库启用了 `production` environment 的 required reviewers，AI agent 必须等待人工审核通过后，才允许 workflow 继续执行正式发布。
- stable 发布完成后，远端 `stable` 分支必须同步到对应 stable tag 的提交。
- stable 发布完成后，`release/vX.Y.Z` 必须合并回 `main`，否则本次发布不算真正收束完成。

## 测试与验证规则

- 新增或修改公共行为时，至少补一类验证：
  - 单元测试
  - 打包/安装 smoke
  - release contract 校验
- 测试中的目录名、模块名、配置值和 proto 路径示例默认使用中性命名；不要把具体业务项目名或业务上下文写进测试。
- 变更打包配置时，必须验证 wheel/sdist 是否包含：
  - `writer/templates/**/*`
  - `hub/templates/**/*`
  - `static/**/*`
- 变更 CLI 或配置行为时，至少跑一轮 `api-doc-server --help`、`api-gen-golang --help`、`api-gen-typescript --help` smoke。
- 变更 release 逻辑时，必须覆盖：
  - `release-version.toml` 解析
  - `_version.py` 同步
  - README / release 文档契约

## AI Agent 提交前自查

- 这次改动属于猜测性修复还是已确认修复？
- 改动是否足够小、足够局部、足够容易回退？
- 是否保留了可验证证据，而不是只留下结论？
- 如果触达用户可见行为，`PRE_README.MD` 和 README 是否已同步？
- 如果触达版本或发布流程，release contract 和测试是否已同步？
